# Lovable Prototyping

## Zweck

Lovable-Prompts für brauchbare klickbare Prototypen erstellen.

## Wann verwenden

Wenn aus einer Idee schnell ein erster oder zweiter UI-Prototyp entstehen soll.

## Inputs

- Produktidee
- Zielgruppe
- gewünschter Freiheitsgrad
- bekannte User-Flows
- spätere Backend-Anforderungen

## Ablauf

1. Entscheiden, ob der Prompt offen oder geführt sein soll.
2. User-Flows, Navigation und Kernseiten beschreiben.
3. Spätere Backend-Erweiterbarkeit berücksichtigen.
4. Optional `DESIGN.md`, `BACKEND.md` und `PRD.md` von Lovable erzeugen lassen.
5. Beim zweiten verbesserten Prototyp Fehler und Reibungspunkte des ersten Prototyps einarbeiten.

## Lovable Prompt V1 offen

Nutze, wenn Lovable viel Freiheit bekommen soll. Beschreibe Ziel, Zielgruppe, Kernproblem, wichtigste Seiten und gewünschte Stimmung.

## Lovable Prompt V1 geführt

Nutze, wenn Seiten, Komponenten, Navigation, Datenzustände und Layout-Anforderungen bereits klar sind.

## Review des ersten Prototyps

Prüfe User-Flows, Navigation, Lesbarkeit, Mobile-Ansicht, Formulare, leere Zustände und falsche Annahmen.

## Lovable Prompt V2 verbessert

Gib Lovable konkrete Korrekturen, priorisierte Reibungspunkte und gewünschte Datei-Ergänzungen.

## Übergabe an Claude Code oder Codex

Gewünschte Dateien:

- `PRD.md`
- `DESIGN.md`
- `BACKEND.md`

Beschreibe danach, welche Teile aus Lovable übernommen, refaktoriert oder neu gebaut werden sollen.

## Übergabe von Lovable an Coding Agent

Nach dem ersten Lovable-Prototyp:

1. Export oder Codebasis lokal öffnen.
2. `DESIGN.md`, `BACKEND.md` und `PRD.md` prüfen.
3. Falsche User-Flows, fehlende Zustände und Designbrüche sammeln.
4. Entweder Lovable Prompt V2 erstellen oder direkt mit Claude Code/Codex weiterarbeiten.
5. Backend-Integration erst planen, wenn die Kernflows im Frontend klar sind.

## Output

- Lovable-Prompt
- empfohlene Dateien
- Hinweise für spätere Weiterentwicklung mit Claude Code oder Codex
